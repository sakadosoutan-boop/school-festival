# 「まちたいむ」運用マニュアル (v6)

対象日：**2026年8月29日(土)・8月30日(日)**
対象者：文化祭実行委員、各ブースの担当者、教職員の運営責任者

---

## 1. このアプリでできること

来場者は「ホーム」で全ブースの待ち時間・混雑・売り切れを、「ステージ」で進行表を、「マップ」で場所を確認できます。各ブースの担当者は、お客さんを案内するたびにボタンを1回押すだけで、待ち時間が自動計算・自動学習されます。

本番運用の原則:

1. **1ブースにつき更新端末は原則1台**(後から操作した内容が優先されるため)
2. お客さんを案内したら**その場で「ご案内しました」を押す**。列が変わったら人数も直す
3. **8分以上放置しない**。画面に黄色いリマインドが出たら必ず更新する
4. アプリの表示より、避難・安全・現地スタッフの案内を優先する
5. 不具合時は紙・ホワイトボード掲示に切り替えられるよう準備する

### PINは2種類

| PIN | 使う人 | できること |
| --- | --- | --- |
| 更新用PIN | 各ブースの担当者 | ブース運用(人数・在庫・営業状態)、ブース追加・編集、ステージ進行の編集 |
| 管理者PIN | 実行委員・システム責任者 | 上記＋全体お知らせ・PIN変更・バックアップ復元・全データ入替 |

**管理者PINはブース担当者へ配らないでください。** 更新用PINが漏れても全データの消去・復元はできません。

## 2. 役割分担

- **システム責任者(1〜2名)**: Supabase・GitHub Pages・PIN管理、スナップショット確認、障害時の復旧判断
- **運営本部(各日2名以上)**: 全体お知らせの発信、ステージ進行のずらし操作、「情報が古い」ブースへの巡回連絡、昼と閉場後のサーバー保存
- **ブース担当(各ブース1名+交代要員)**: ご案内ボタン・人数・在庫の更新、端末とバッテリーの管理
- **巡回担当**: 長時間更新のないブースの現地確認

## 3. 事前準備

### 3.1 バックエンド

READMEの手順でSupabase(マイグレーション001→002→003)とEdge Function、GitHub Secretsを設定します。デモモードのまま公開すると画面に常時警告が出ます。

本番前チェック:

- スマホAで一覧が見える／スマホBの更新が30秒以内にAへ反映される
- 更新用PIN・管理者PINを初期値(202608 / 202609)から変更済み
- 更新用PINの端末に「全体へのお知らせ」「PINの変更」が**表示されない**
- 「デモ表示中」の警告が出ていない
- iPhoneで「ホーム画面に追加」してアイコンが表示される

### 3.2 ブース登録

**参加43団体とステージ12公演は初期データとして用意済みです。** 本番サイトの初回だけ、管理者ログイン → 設定 →「読み込む」で `yanagi2026-import.json`(公開サイトの `/yanagi2026-import.json` をダウンロード)を取り込んでください。以降の追加・修正はスタッフタブ →「新しいブースを追加」「編集」で行います(CSVは廃止。全てアプリ内で完結)。

- **アイコン**: 絵文字か写真(自動で正方形に縮小)
- **運営団体**: クラスは学年×組をドラムロールで選択 →「2年1組」表記に自動統一
- **場所**: 棟(HR棟/特別教室棟/管理棟/野外) × 階 × 教室名
- **マップに点灯させるコツ**: クラス企画は自動でホームルーム教室(例: 2年6組→「2-6」)に点灯します。特別教室の場合は教室名をマップの表記どおり(「視聴覚室」「301」など)に入力してください
- **1回に同時案内できる人数**: 待ち時間計算の要。たこ焼き5人前同時なら5、脱出ゲーム4人1組なら4

登録後、「全体バックアップ(書き出す)」と管理者の「サーバーへ保存」で控えを作ります。

### 3.3 ステージ登録

スタッフタブ →「ステージ進行を管理」で、日ごとの公演(タイトル・出演者・開始/終了・お知らせ)を登録します。

### 3.4 端末準備

- ブースごとに更新用スマホ1台 + 予備、モバイルバッテリー
- 「ホーム画面に追加」を済ませ、公開URLと運営本部連絡先を書いた紙を配布
- 通信不能時に掲示する「現在約○分待ち」の紙 or ホワイトボード
- 裏に回している(他アプリ・画面消灯)間は自動更新が止まり、開き直すと即時更新されます。操作前に1〜2秒待つこと

## 4. リハーサル(前日までに30分)

1. 更新用PINでログインし、担当ブースで「ご案内しました」→ 人数が減り、待ち時間が変わることを確認
2. 押し間違い →「直前の操作を取り消す」(1分以内)で戻ることを確認
3. 商品を登録し、在庫を0にする → 来場者画面に「売り切れ」が出ることを確認
4. 機内モードで人数を更新 → 「未送信」の警告 → 通信を戻すと自動送信されることを確認
5. ステージで「+5分ずらし」を練習(終了済みは動かない)
6. 更新用PINの端末に管理メニューが出ないことを確認
7. 管理者で「サーバーへ保存」→ 1件変更 →「保存履歴・復元」で戻す練習
8. 全体お知らせの公開と解除を練習

## 5. 当日の標準手順

### 開場前

- 運営本部: 公開サイト確認、全体お知らせが古くないか確認、「サーバーへ保存」を実行
- ブース担当: ログインして担当ブースを開き、「準備中」のまま列の想定人数を0に。開店したら「🟢営業中」へ切替

### 営業中

- 1組案内するたびに「ご案内しました」(1組あたりの所要時間を自動学習し、精度が上がります)
- 列の実数と表示がずれたら人数を直接修正(5/10/20/50のクイックボタン)
- 食品は在庫を減らしていき、0で自動「売り切れ」。手動の「売り切れに」も可
- 休憩・安全確認は「🔴準備中」へ。再開したら「🟢営業中」
- 黄色い「○分間更新されていません」が出たら、実際の列を見て更新
- 迷ったら「待ち時間 計算ガイド」で3ステップ入力

### ステージ進行(運営本部)

- 押し/巻きが出たら「ステージ進行を管理」→ ±5〜15分でまとめてずらす(終了済みの公演は動きません)
- 中止は公演ごとに「中止」→ 来場者には打ち消し線+中止バッジで表示。「再開」で戻せます
- 変更・注意事項は公演の「お知らせ」欄へ(来場者のタイムテーブルに黄色で出ます)

### 緊急時(運営本部・管理者)

- 設定 →「全体へのお知らせ」に入力して公開 → 全員の画面に赤帯で表示
- 解消したら**必ず空にして解除**
- 放送・現地掲示・誘導を併用。アプリだけに頼らない

### 閉場後

- 全ブースを「準備中」へ
- 管理者が「サーバーへ保存」と「書き出す」(JSON)を実行

## 6. 障害別の対応

### 6.1 「情報が古い」「更新待ち」と出る

12分以上更新がないと表示され、30分で待ち時間の数字が隠れます。担当者へ連絡し、現地の列を数えて更新してもらいます。担当者不在なら運営本部が代理更新します。

### 6.2 電波が悪い・オフライン

- 更新は端末に保留され、回復後に自動送信されます(スタッフメニューに未送信件数を表示)
- 来場者画面は最後に取得した情報を表示し続けます
- 復旧しない間は紙・ホワイトボード掲示に切り替えます

### 6.3 データが消えた・おかしくなった

1. 操作を止める
2. 管理者PINでログイン → 設定 →「保存履歴・復元」
3. 「入替前の自動保存」など直前のスナップショットを選び「この時点へ復元」
4. スナップショットが使えない場合は、書き出したJSONを「読み込む」で復元

ブースを誤って削除した場合、削除内容はサーバーの監査ログ(audit_log)に丸ごと残っています。システム責任者がSupabaseで確認・復旧できます。

### 6.4 PINを忘れた

更新用PINは管理者がアプリの設定から再設定できます。管理者PIN自体を忘れたら、SupabaseのSQL Editorで `select set_admin_pin('新しいPIN');` を実行します。

PINの連続失敗は端末ごと(10分8回で15分ロック)と全体(10分40回で10分停止)で制限されます。ロック中は正しいPINでも待つ必要があります。

### 6.5 画面が真っ白・エラーカードが出た

エラー画面の「保存データをリセットして再読み込み」を押します(この端末の表示用データを消すだけで、サーバーの共有データは消えません)。直らなければ別ブラウザ・予備端末で開きます。

### 6.6 編集しても保存されない(操作すると「保存できませんでした」と出る・少し経つと元に戻る)

サーバー側の更新処理が失敗しています。画面に出るエラー文をそのまま確認してください。

- **「データベースに未実行のマイグレーションがあります…」** → Edge Functionだけ新しく、データベースのマイグレーションが追いついていない状態です。SupabaseのSQL Editorで `supabase/migrations/` のSQLを番号順に実行すれば、その場で直ります(どれも再実行して安全です)
- **「別の端末で先に更新されました」** → 故障ではありません。最新の値が自動で読み込まれるので、確認してからもう一度操作します
- **それ以外** → Supabaseの Edge Functions → Logs でエラー内容を確認し、システム責任者が対応します

Edge Function(`super-handler`)は、`supabase/` 配下の変更がmainに入ると GitHub Actions(Supabase Backend)が自動でデプロイします。このとき004のマイグレーションが未適用だと自動デプロイは中止されるので、先にSQL Editorでマイグレーションを実行してください。手動でデプロイする場合も同じ順番です(README参照)。

## 7. 安全と個人情報

- 氏名・電話番号・顔写真など個人を特定できる情報をブース紹介やアイコン画像に入れない
- 待ち人数は概数。列が危険な場合は受付を止め、通路確保を優先する
- 食品のアレルギー情報はアプリに頼らず、必ず現地掲示を用意する
- PIN・管理画面のスクリーンショットをSNSへ投稿しない

---

## 8. バックアップ規則

次の時点で管理者が「サーバーへ保存」+「書き出す」(JSON)の両方を実行します。

- 前日: 登録完了後
- 両日: 開場前・12時頃・閉場後

サーバー側スナップショットは最新30件を自動保持し、全データ入替・復元の直前にも自動保存されます。JSONファイルは共有ドライブにも複製し、来年への引き継ぎに使います。
