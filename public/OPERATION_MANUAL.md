# 「まちたいむ」運用マニュアル

対象日：**2026年8月29日（土）・8月30日（日）**  
対象者：文化祭実行委員、各企画の高校生担当者、教職員の運営責任者  
対象利用者：来場者・生徒

---

## 1. このアプリでできること

「まちたいむ」は、来場者が企画の場所・内容・待ち時間とステージ等のタイムテーブルを確認するためのWebアプリです。各企画の担当者は、スマートフォンから列の人数、営業状況、売切・受付終了、臨時のお知らせを更新します。

本番運用では、次の原則を守ってください。

1. **1企画につき更新端末は原則1台**にします。
2. 待ち時間は最低でも**5〜8分に1回**確認します。
3. アプリの表示より、避難・安全・現地スタッフの案内を優先します。
4. 全件置換や大きな変更の前には、必ずバックアップを保存します。
5. 不具合が起きても、受付停止や紙掲示に切り替えられるよう準備します。

---

## 2. 役割分担

### システム責任者（教職員または実行委員1〜2名）

- Supabase、GitHub Pages、環境変数を管理する
- 本番前のデータ取込とバックアップを行う
- スタッフPINを変更し、必要な人だけに共有する
- 障害時の復旧判断を行う

### 運営本部データ担当（各日2名以上）

- 朝の稼働確認を行う
- タイムテーブル変更、企画中止、緊急告知を反映する
- 「更新が古い企画」を巡回担当へ連絡する
- 12時頃と閉場後にバックアップを保存する

### 各企画の更新担当（各企画1名＋交代要員1名）

- 担当企画の画面を開いた状態にする
- 列の人数、営業状況、お知らせを更新する
- 端末交代時に現在値と注意事項を引き継ぐ
- バッテリー残量と通信状況を確認する

### 巡回担当

- 長時間更新されていない企画を現地確認する
- 端末故障、担当者不在、列の混乱を運営本部へ報告する

## 3. 事前準備

### 3.1 共有バックエンドの設定

デモモードは同じ端末内でしかデータが共有されません。本番では、READMEの手順に従ってSupabase Edge Functionを設定してください。

本番確認項目：

- 来場者用スマートフォンAで企画一覧が見える
- スタッフ用スマートフォンBで列人数を更新する
- Aの表示が10〜20秒以内に変わる
- スタッフPINが初期値 `202608` から変更されている
- サービスロールキーがブラウザ側やGitHubに入っていない
- 公開URLをシークレットウィンドウで開ける

### 3.2 データファイルの作成

運営画面の「企画テンプレート」「時間割テンプレート」からCSVを取得し、ExcelまたはGoogleスプレッドシートで編集します。

#### 企画・団体一覧の必須項目

- `id`：英小文字・数字・ハイフン・アンダースコア。年度内で重複しない固定値
- `name`：企画名
- `category`：指定された英字カテゴリ
- `location`：来場者が迷わない表記
- `days`：`2026-08-29`、`2026-08-30`。両日なら `|` で区切る
- `open_time`、`close_time`：24時間表記の `HH:MM`
- `capacity`：1回に案内できる人数
- `cycle_minutes`：1回の案内にかかる分数
- `status`：準備データは原則 `closed`

`id` は途中で変えないでください。同じidの行を再取込すると、その企画が更新されます。idを変えると別企画として追加されます。

#### タイムテーブルの必須項目

- `id`
- `day`
- `start_time`、`end_time`
- `title`
- `venue`

同じ会場で時間が重なる場合、取込確認画面に警告が出ます。転換時間を含めて確認してください。

### 3.3 取込手順

1. 運営タブからスタッフPINでログインします。
2. 現在のデータを「全体バックアップ」で保存します。
3. 「企画・団体一覧を取込」または「時間割を取込」を押します。
4. CSVまたはJSONを選びます。
5. エラーと警告を確認します。
6. 通常は「追加・更新」を使用します。
7. ファイルにないデータを削除したいときだけ「全件置換」を使用します。削除確認チェックを入れた後、反映直前にアプリが自動バックアップを保存します。
8. 反映後、来場者画面で検索・日付切替・表示順を確認します。

**エラーが1件でもある場合は反映できません。** 警告は反映できますが、理由を確認してから進めてください。

### 3.4 端末準備

各企画につき以下を用意します。

- 更新用スマートフォン1台
- 交代用または予備スマートフォン1台（複数企画で共用可）
- モバイルバッテリーと充電ケーブル
- 担当企画名・公開URL・運営本部連絡先を書いた紙
- 通信不能時に掲示する「現在約○分待ち」「受付停止」の紙またはホワイトボード

ブラウザの「ホーム画面に追加」を行い、公開URLを探さなくても開けるようにします。

## 4. リハーサル

本番前日までに、実際の担当者で30分以上の通し練習をします。

1. 全企画を `closed` にします。
2. 3企画以上を同時に `open` にします。
3. 列人数を増減し、来場者画面への反映を確認します。
4. 「○人をご案内しました」を押して人数が減ることを確認します。
5. 2台で同じ企画を同時更新し、競合警告を確認します。
6. 機内モードで更新し、未送信表示を確認します。
7. 通信を戻し、保留更新が再送されることを確認します。
8. 誤ったCSVを読み込み、エラーで止まることを確認します。
9. バックアップJSONを保存し、ファイルが開けることを確認します。
10. 企画中止、売切、会場変更を想定して連絡経路を練習します。

## 5. 当日の標準手順

### 5.1 開場60分前

システム責任者：

- 公開サイトを通常ブラウザとシークレットウィンドウで確認
- 8月29日／30日の両方へ切り替えられることを確認
- 企画一覧、タイムテーブル、緊急告知を確認
- バックアップを保存し、ファイル名に日付と時刻を付ける
- スタッフPINでログインできることを確認

各企画担当：

- 担当企画を選択
- 場所、団体名、案内人数、1回の所要時間を確認
- 開場までは `closed` にする
- バッテリー80%以上、モバイルバッテリー接続を確認

### 5.2 開場時

- 実際に受付を開始してから `open` にします。
- 初回の列人数を数え、入力します。
- 待ち時間が不自然な場合は、`capacity` と `cycle_minutes` の設定を運営本部に確認します。

### 5.3 営業中

- 列の増減が大きいときは都度更新します。
- 変化がなくても5〜8分ごとに確認します。
- 1組を案内したら「○人をご案内しました」を押します。
- 整理券配布、入口変更、休憩、材料不足は「お知らせ」に短く記載します。
- 休憩・安全確認・一時停止は `paused` にします。
- 売切、整理券終了、最終受付終了は `sold_out` にします。
- 終了後は `closed` にします。

### 5.4 交代時

交代前に次を口頭で伝えます。

- 現在の列人数
- 次回案内のタイミング
- 整理券や売切の状況
- 端末の充電残量
- 直近のトラブル

交代後の担当者は、自分で画面の現在値を確認してから引き継ぎ完了とします。

### 5.5 閉場後

- 全企画を `closed` にします。
- 運営本部が全体バックアップを保存します。
- 翌日の変更がある場合はCSVを修正し、「追加・更新」で反映します。
- 端末を充電し、モバイルバッテリーを回収します。

## 6. 障害別の対応

### 6.1 来場者画面に「情報が古い」と出る

原因：12分以上更新されていません。30分以上で待ち時間の数字を隠します。

対応：

1. 現地へ連絡し、現在の列人数を確認する
2. 担当者が更新できない場合、運営本部が代理更新する
3. 長時間復旧しない場合、現地確認を優先する旨を案内する

### 6.2 インターネットにつながらない

来場者画面は最後に取得したデータを表示します。スタッフの更新は端末に保留され、通信復旧後に再送を試みます。

対応：

1. Wi-Fiとモバイル回線を切り替える
2. 機内モードを一度オン・オフする
3. ブラウザを再読み込みする
4. 復旧まで紙・ホワイトボードで待ち時間を掲示する
5. 保留送信が競合した場合、現地の最新人数を数え直して更新する

### 6.3 同じ企画を2台で操作し、競合警告が出た

アプリは、古いリビジョンからの上書きを拒否します。

対応：

1. 警告が出た端末で最新値を読み込む
2. 現地の列人数を数え直す
3. 更新担当端末を1台に決める
4. 必要なら運営本部へ報告する

### 6.4 CSVが取り込めない

- Excelで「CSV UTF-8（コンマ区切り）」として保存したか確認
- 見出し名を変更していないか確認
- `id` が重複していないか確認
- 時刻が `9:30` ではなく `09:30` になっているか確認
- 開催日が `2026-08-29` または `2026-08-30` になっているか確認
- セル内改行や引用符が多い場合は、一度テキストを簡略化する

### 6.5 全件置換後に企画が消えた

1. 新たな取込操作を止める
2. 直前に保存したバックアップJSONを確認する
3. バックアップJSONを企画一覧またはタイムテーブルとして読み込む
4. 「全件置換」で復元する
5. 復元後、来場者画面を確認する

バックアップがない場合、Supabaseのバックアップ機能またはCSV原本から復元します。

### 6.6 PINを忘れた

PINは来場者向けサイトから確認できません。システム責任者だけが再設定を行います。PINをチャットの公開チャンネルや校内掲示に書かないでください。

### 6.7 端末が故障・電池切れ

1. 予備端末で公開URLを開く
2. スタッフPINでログインする
3. 担当企画を選ぶ
4. 現地の列人数を数え直して更新する
5. 故障端末の保留更新が後から送信されないよう、可能ならブラウザを閉じる

### 6.8 重要なお知らせを更新できない

重要なお知らせは、通信復旧後に古い内容が遅れて公開される事故を防ぐため、オフライン時には保留送信しません。

1. 通信状態を確認する
2. 別回線または運営本部の端末から再実行する
3. 公開後は来場者画面で文面と表示時刻を確認する
4. 状況が解消したら必ず「表示を解除」する
5. 緊急時はアプリだけに頼らず、放送・校内掲示・現地誘導を併用する

### 6.9 表示が真っ白、操作不能

1. ページを再読み込みする
2. ブラウザを完全に閉じて開き直す
3. 別ブラウザまたは予備端末で開く
4. 公開URLが正しいか確認する
5. GitHub Pagesのデプロイ状況とSupabase Functionログをシステム責任者が確認する

## 7. 安全と個人情報

- 氏名、電話番号、顔写真、個人を特定できる情報をCSVへ入れないでください。
- 来場者数・待ち人数は概数です。安全な通路幅を優先し、列が危険な場合は受付を止めます。
- 緊急時はアプリ更新より放送、避難誘導、教職員の指示を優先します。
- 食品企画のアレルギー情報はアプリだけに依存せず、必ず現地掲示を用意します。
- PIN、サービスロールキー、管理画面のスクリーンショットをSNSへ投稿しないでください。

## 8. バックアップ規則

最低限、次の時点でJSONバックアップを保存します。

- 8月28日：最終データ確定後
- 8月29日：開場前、12時頃、閉場後
- 8月30日：開場前、12時頃、閉場後
- 全件置換の直前
- 大規模なタイムテーブル変更の直前

推奨ファイル名：

```text
machitime-backup-2026-08-29-0900.json
machitime-backup-2026-08-29-1200.json
```

バックアップは、システム責任者のPCと共有ドライブの2か所に保存します。

## 10. 終了後の引き継ぎ

文化祭終了後、以下を保存してください。

- 最終バックアップJSON
- 実際に使用した企画CSVとタイムテーブルCSV
- トラブルと対応の記録
- 更新頻度、通信状況、端末台数の振り返り
- 来年度に変えるべき項目

PINは翌年度に再利用せず、データも個人情報が残っていないことを確認してから保管します。
