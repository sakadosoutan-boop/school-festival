# まちたいむ — 文化祭 待ち時間アプリ (v6 / プロトタイプv4系)

2026年8月29日(土)・30日(日)の文化祭で、来場者と生徒がブースの待ち時間・在庫、ステージ進行、会場マップを確認するためのPWAです。プロトタイプv4(最新版)のUI・機能・データモデルをそのまま採用し、その下に複数端末で同期する共有API(Supabase)、権限分離、バックアップ、負荷対策などの実運用インフラを接続しています。

## 主な機能

- **ホーム**: 待ち時間(自動計算)・混雑ステータス・推移グラフ・トレンド、カテゴリ絞込、検索(ひらがな/カタカナ/全角半角を同一視)、並び替え、お気に入り、完売表示、情報鮮度(12分で「更新待ち」・30分で数字を非表示)
- **ステージ**: ロッキン風タイムテーブル(現在時刻ライン・上演中ハイライト)とリスト表示、NOW/NEXT、2日分切替。各公演はブースと同じくアイコン(絵文字/画像)・紹介文を持ち、タイムテーブルや下の出演団体一覧をタップすると詳細が開く
- **マップ**: 校内図に忠実なSVGマップ。企画のある教室が点灯し待ち時間を表示、タップで詳細へ。食品🍴・自販機🥤・ゴミ箱🗑️
- **スタッフ**: PINログイン → 担当ブースを「運用する」。ご案内ボタン(1組あたり時間を自動学習)、1分以内の取り消し、人数調整、計算ガイド、商品在庫(残りわずか/売り切れ)、8分間未更新のリマインド、ブースの追加・編集・削除、ステージ進行の編集(公演ごとのアイコン・紹介文、まとめて±5〜15分ずらし・中止)
- **管理者**: 全体お知らせ(全画面に表示)、落とし物・迷子の掲示板、一斉開店・一斉閉店、PIN変更(更新用/管理者)、JSONバックアップ読込、サーバー側スナップショットの保存・一覧・ワンタップ復元、サンプルリセット
- **アレルギー表示**: 商品ごとに特定原材料8品目を選ぶと、来場者の画面に⚠表示(目安表示。現地掲示が正)
- **QR・共有リンク**: `?b=ブースID` で詳細直行、`?tab=stage|map` でタブ指定。ブース詳細の「共有」ボタンからURLを取得できます
- **投票**: `VITE_VOTE_FORM_URL` を設定するとホームに投票バナーを表示(未設定なら非表示)
- オンボーディング、ヘルプFAQ、白画面ゼロのエラー復旧カード、PWA(iOS向けPNGアイコン同梱)

## 同期と負荷対策(実運用向け)

- ブースは1件ずつ独立して保存され(書込衝突なし・後勝ち)、通信断の更新は端末に保留して回復後に自動送信
- ポーリングは共有APIではスタッフ約12秒/来場者約25秒＋ゆらぎ。失敗時は間隔を自動延長、裏のタブは停止し復帰時に即時更新
- ETag方式で変更がない間はペイロード(アイコン画像含む)を転送しない
- 更新時刻はサーバーが刻む(端末の時計ズレで鮮度表示が壊れない)
- サーバー側でも全入力を検証・サイズ制限し、待ち時間を再計算

## ローカル起動

```bash
npm install
cp .env.example .env.local
npm run dev
```

環境変数が空の場合は**デモモード**(この端末内だけで動作)。初期PINは更新用 `2025` / 管理者 `202609`。本番ビルドでデモのまま公開すると画面上部に常時警告が出ます。

### 初期データ(やなぎ祭2026)

参加団体一覧・体育館当日スケジュール・校内マップの実データを同梱しています。

- デモモードは**参加43団体・ステージ12公演**が最初から入った状態で起動します(全ブース準備中)
- 本番(Supabase)は空で始まるため、管理者ログイン → 設定 → 読み込む で `public/yanagi2026-import.json`(公開サイトの `/yanagi2026-import.json` からもダウンロード可)を取り込んでください
- データの修正は `src/lib/yanagi2026.ts` を編集し、`npm run data:json` でJSONを再生成します。アプリ内の編集画面から直しても構いません

### PINと権限

| 権限 | できること | デモ初期PIN | 本番初期PIN |
| --- | --- | --- | --- |
| 更新用(スタッフ) | ブース運用・追加・編集・削除、ステージ編集、バックアップ書き出し | `2025` | `202608` |
| 管理者 | 上記＋全体お知らせ・PIN変更・JSON読込・スナップショット復元・リセット | `202609` | `202609` |

PINは4〜8桁。更新用PINが漏れても、全データ入替や復元・PIN変更はできません。**本番前に必ず両方変更してください。**

## 本番用バックエンド(Supabase)

1. Supabaseプロジェクトを作成
2. SQL Editorで `supabase/migrations/` の `001_init.sql` → `002_hardening.sql` → `003_v4_model.sql` → `004_operational_hardening.sql` → `005_notice_board.sql` を順に実行(旧スキーマに入力済みデータがあれば003が自動変換します)
3. Edge Functionを反映

> ⚠️ **Edge Functionのコードとマイグレーションはセットで更新してください。** 現行の `super-handler` は004で作られるRPC(楽観ロック・原子的レート制限)を呼びます。004未実行のままFunctionだけ更新すると、更新系がすべてエラーになります。また `ALLOWED_ORIGIN` に `*` は設定できません(起動時に拒否されます)。ローカル検証する場合は `https://...github.io,http://localhost:5173` のように明示してください。

```bash
supabase link --project-ref YOUR_PROJECT_REF
supabase db push
# 複数オリジンはカンマ区切り(本番 + ローカル検証など)
supabase secrets set ALLOWED_ORIGIN=https://YOUR_ACCOUNT.github.io
supabase functions deploy super-handler
```

4. GitHubリポジトリのActions secretsへ設定:

```env
VITE_FESTIVAL_API_URL=https://YOUR_PROJECT.supabase.co/functions/v1/super-handler
VITE_FESTIVAL_PUBLIC_KEY=YOUR_PUBLISHABLE_KEY
# 任意: 投票フォーム
VITE_VOTE_FORM_URL=https://forms.gle/xxxx
# 任意: SupabaseのPersonal Access Token。設定すると supabase/ 配下の変更がmainに
# 入るたびに、004適用チェック → Edge Function自動デプロイ → 疎通確認まで実行される
SUPABASE_ACCESS_TOKEN=sbp_xxxx
```

`SUPABASE_ACCESS_TOKEN` を設定した場合、Actionsの「Supabase Backend」が **004未適用ならデプロイを中止**するため、「Functionだけ新しくて書込が全部失敗する」事故を防げます。未設定の間は警告だけ出してスキップします(手動デプロイ運用)。

Secretsが未設定のままだとデプロイは失敗して止まります(誤ってデモモードを本番公開しないため)。意図的にデモを公開する場合だけ、リポジトリ変数 `ALLOW_DEMO_DEPLOY=true` を設定してください。

テーブルはすべてRLS有効で、ブラウザからの直接アクセスは不可。読み書きはEdge Function経由のみです。サービスロールキーをフロント側へ置かないでください。

## セキュリティ・データ保全

- PIN認証は端末単位(10分8回で15分ロック)と全体(10分40回で10分停止)の二段レート制限
- スタッフの全書込は `audit_log` に記録(IP+UAのハッシュのみ)。**ブース削除時は消えたデータ全体をログに残す**ため、誤削除も復旧材料が残ります
- 全データ入替・復元の直前は必ずサーバー側スナップショットを自動保存(最新30件保持)。管理者は設定画面からワンタップで復元できます
- JSONバックアップにはPINを含めません
- アイコン画像は端末側で256px JPEGへ縮小し、サーバー側でも形式・サイズを検証します
